OVERVIEW

What is the Salesforce Org Comparator?

The Salesforce Org Comparator is a reusable DevOps tool built by the Salesforce GDC Team. It provides a side-by-side metadata diff between two Salesforce orgs, enabling teams to identify drift, perform safe back-promotions, and validate changes — all without leaving their local environment.

Key capabilities:

  • Side-by-side XML diff — compare any metadata type between two orgs
  • Back-promotion PR creation — raises a GitHub PR from source to target, no direct deployment
  • Checkonly validation — runs a checkonly deploy to the target org before any merge
  • package.xml export — download a ready-to-use package manifest of selected components
  • AI conflict resolution — Claude AI proposes resolutions for XML conflicts, with explicit user approval gates
  • Demo mode — try all features with mock data, no real org required
PRE-REQUISITES

What you need before starting

  • ✅ Node.js 18+ — Download from https://nodejs.org. Verify with node -v.
  • ✅ Python 3.8+ — Download from https://python.org. Verify with python3 --version.
  • ✅ Salesforce CLI — Install with npm install -g @salesforce/cli. Verify with sf --version.
  • ✅ At least one authenticated SF org — Run sf org login web --alias myOrg to authenticate. Verify with sf org list.
  • ✅ GitHub account — Required only for the back-promotion PR feature. A personal access token with repo scope is needed.
  • ℹ No org credentials needed for Demo Mode — toggle Demo Mode in the UI header to explore all features with mock data.

INSTALLATION

Installation & Starting the Tool

Step 1 — Clone the repository:

# Clone from GitHub
git clone https://github.com/navjeetsalesforce/sf-org-comparator

# Move into the project folder
cd sf-org-comparator

Step 2 — Start both backend and frontend with one command:

# Make the start script executable, then run it
chmod +x start.sh && ./start.sh

The script starts the Python FastAPI backend on port 8000 and the React frontend on port 5173 automatically.

Step 3 — Open the tool in your browser:

http://localhost:5173
Tip: Keep the terminal running — closing it will stop both servers. Open a second terminal tab if you need to run other commands.
AUTH

Authenticating your Salesforce Orgs

Before comparing orgs, authenticate each org using the Salesforce CLI. Run each command in a terminal — a browser window will open for OAuth login:

# Authenticate your Production org
sf org login web --alias PROD

# Authenticate your UAT / staging org
sf org login web --alias UAT

# Verify both orgs are listed
sf org list

Once authenticated, the org aliases (e.g. PROD, UAT) will appear in the dropdown menus inside the tool.

DEMO MODE

Trying the tool without real orgs

Toggle Demo Mode using the switch in the top-right of the UI header. This loads pre-built mock data so you can explore all comparison, diff viewer, and back-promotion features without connecting to a live Salesforce org.

Note: Demo Mode data is entirely fictional. PR creation and validation actions in Demo Mode do not connect to GitHub or Salesforce.

COMPARISON

Running an Org Comparison

Follow these steps to compare metadata between two orgs. The comparison fetches live metadata from both orgs via the Salesforce CLI and displays a structured diff in the browser.

  1. Select Source Org — Choose your source org (e.g. Production) from the Source dropdown. Only authenticated orgs appear here.
  2. Select Target Org — Choose your target org (e.g. UAT) from the Target dropdown.
  3. Choose metadata types — Select one or more metadata types to include in the comparison: ApexClass, Flow, Profile, CustomObject, Layout, PermissionSet, and others. Use the checkboxes to filter the scope.
  4. Click "Compare" — The backend retrieves metadata from both orgs via SFDX and computes the diff. A progress indicator is shown during retrieval.
  5. Review the summary bar — Once complete, the summary bar shows four counts: Identical, Different, Source Only, and Target Only.
  6. Filter results — Use the search bar above the results table to filter by component name or API name.
  7. Open the XML diff viewer — Click any row in the results table to open a side-by-side XML diff for that component.
DIFF VIEWER

Reading the XML Diff

The XML diff viewer uses colour-coded highlighting to indicate the type of change between the source and target versions of a component:

Added (present in Source, missing in Target)
Removed (missing in Source, present in Target)
Changed (value differs between Source and Target)
Identical (no difference on this line)
Result StatusMeaning
Identical Component XML is byte-for-byte the same in both orgs. No action needed.
Different Component exists in both orgs but the XML content differs. Click the row to see what changed.
Source Only Component exists in the source org but not in the target. Likely a new addition.
Target Only Component exists in the target org but not in the source. May have been deleted from source or added directly to target.
Tip: Use the package.xml export button to download a package.xml manifest of all selected (ticked) components. This can be used directly with SFDX or Ant migration tooling.

BACK PROMOTION

Back Promotion Workflow

The back-promotion workflow creates a GitHub Pull Request containing the selected metadata changes. It never deploys directly — all deployments must be done manually after the PR is reviewed and validated.

  1. Tick components to back-promote — In the comparison results table, use the checkboxes on the left to select one or more components you want to promote back.
  2. Click "Create Back-Promotion PR" — The button appears in the Action Panel on the right-hand side once at least one component is selected.
  3. Enter your GitHub repository name — Provide the owner/repo reference (e.g. myorg/salesforce-repo) and your GitHub personal access token when prompted.
  4. PR is created — no deployment happens — The tool commits the selected metadata XML files to a new branch and opens a PR from source branch to target branch on GitHub. Nothing is deployed to Salesforce at this stage.
  5. Review the PR on GitHub — Open the PR URL provided. Inspect the file diff, review comments, and confirm the changes are correct.
  6. If conflicts exist — AI Conflict Resolution is triggered — See Page 5 for the full approval-gated AI resolution workflow.
  7. Approve the PR on GitHub — Once you are satisfied with the changes, approve the PR via GitHub's review flow.
  8. Run "Validate on Target" (checkonly deploy) — Click this button in the tool to run a checkonly deploy to the target org. This runs Apex tests and validates the metadata without committing any changes to the org.
  9. Merge the PR manually on GitHub — Only after validation passes, return to GitHub and merge the PR yourself. The tool does not auto-merge.
⚠
Approval Gates — Back Promotion Workflow
  • The tool never deploys metadata directly to any Salesforce org at any stage of this workflow.
  • The PR must be reviewed and approved by a team member on GitHub before any merge.
  • A checkonly validation must pass before the PR is eligible to be merged.
  • The PR must be merged manually by the user on GitHub — the tool does not auto-merge under any circumstances.

AI RESOLUTION

AI Conflict Resolution (with Approval Gates)

When a back-promotion PR contains XML merge conflicts, the tool can invoke Claude AI to propose a resolution for each conflicted file. Every proposed resolution requires explicit user approval before it is committed — the AI never modifies files autonomously.

How it works — step by step:

  1. Conflicts are detected — When the back-promotion PR has one or more XML merge conflicts, a notification appears in the tool's Action Panel. Click "Resolve Conflicts with AI" to begin.
  2. Claude AI analyses each conflict — The tool sends each conflicted Salesforce XML file to the Claude AI engine. The AI examines both versions of the file and the surrounding context.
  3. For each file, the tool presents:
    • The conflict displayed as both versions side by side
    • The AI's proposed resolution with a plain-English explanation of the decision
    • A diff preview showing exactly what would change if the resolution is applied
  4. You approve or reject each file individually — For every conflicted file, you must click either "Approve" or "Reject". There is no bulk-approve option.
  5. Rejected files remain as conflicts — Any file you reject stays in a conflicted state and must be resolved manually in your local Git client or directly on GitHub.
  6. Approved resolutions are committed to the PR branch — The approved file content is committed to the back-promotion branch. This does not deploy anything to Salesforce.
  7. Validation runs automatically after all approvals — Once you have processed all files (approved or rejected), a checkonly validation deploy is triggered automatically against the target org.
  8. You merge the PR manually on GitHub — Only after validation passes and you are satisfied with all resolutions, return to GitHub to merge the PR yourself.
⚠
Approval Gates — AI Conflict Resolution
  • The AI never deploys anything directly. It only proposes resolutions to XML conflicts in the PR branch.
  • Every resolution requires explicit user approval. Each file must be individually approved or rejected — there is no auto-accept.
  • Rejected files are not modified. They remain as conflicts for the user to handle manually.
  • Approved resolutions are only committed to the PR branch — not deployed to any org.
  • Checkonly validation runs before any merge to confirm the resolved XML is deployable.
  • The PR must be manually merged by the user on GitHub. The AI cannot and does not trigger a merge.
Best practice: Always read the AI's explanation for each resolution before approving. For complex Profiles or Permission Sets, the AI may flag that it is uncertain — treat these as high-priority manual reviews.

TROUBLESHOOTING

Common Issues & Fixes

ProblemLikely CauseSolution
Backend won't start Python version too old or pip3 missing Run python3 --version — must be 3.8 or above. Install pip3 with python3 -m ensurepip --upgrade. Then re-run ./start.sh.
No orgs showing in dropdown No authenticated orgs found by SFDX Run sf org list in a terminal. If empty, authenticate with sf org login web --alias myOrg and refresh the browser.
Comparison fails mid-run Org session expired The SFDX session token has expired. Re-authenticate the affected org: sf org login web --alias PROD, then re-run the comparison.
CORS error in browser console Backend not running on port 8000 Check that ./start.sh is still running. Open http://localhost:8000/health — you should see {"status":"ok"}. Restart ./start.sh if needed.
PR creation fails GitHub token missing or insufficient scope The personal access token must have the repo scope (full repository access). Re-generate the token in GitHub → Settings → Developer settings → Personal access tokens.
Validation returns errors for unchanged files Org dependencies or missing components The target org may be missing dependent metadata. Review the error detail in the validation results panel. Add the required dependent components to the selection before creating the PR.
Frontend shows blank page on load Build artefacts missing or Vite not running Ensure ./start.sh completed without errors. Check the terminal for the exact URL — Vite may have assigned a different port (e.g. 5174) if 5173 was in use.
Want to test without real orgs No authenticated orgs available Toggle Demo Mode using the switch in the top-right of the UI header. All comparison, diff, and back-promotion screens are available with mock data.
